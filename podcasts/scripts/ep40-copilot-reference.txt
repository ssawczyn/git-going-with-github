[ALEX]
That's a great analogy, Jamie! You can think of them like specialized researchers you're directing. When you type an "at" symbol, `@`, in the chat input, you're telling Copilot to pull context from a specific source.

[ALEX]
For example, `@workspace` tells Copilot to search your entire VS Code workspace for relevant information. It's like telling your research assistant, "Look through all the files in our current project for an answer."

[JAMIE]

[JAMIE]

[ALEX]
Exactly. And for those times you're working in the integrated terminal in VS Code, there's `@terminal`. This provides context from your terminal, which is super useful for debugging. You could ask, "At terminal, what did the last command output mean?"

Beyond those broader participant scopes, we also have what are called "variables," which are even more specific. You access these by typing a hash symbol, `#`.

[ALEX]

[JAMIE]

[ALEX]
Speaking of custom instructions, this is where Copilot really shines for tailoring its behavior to your specific needs or your team's standards. Copilot supports multiple ways to provide these instructions, each with different scopes and priorities.

[JAMIE]
Okay, so it's not just one file where I dump all my rules?

[ALEX]
No, it's more sophisticated than that. The primary way to provide project-wide instructions is through a file called `.github/copilot-instructions.md`. This file lives at the root of your repository, inside the `.github` folder. Its content is automatically included in *every* chat request and inline suggestion context for that project. You never need to invoke it manually.

[JAMIE]
So, this is like the official style guide or rulebook for *this specific project* that Copilot always refers to?

[ALEX]
Exactly. You'd put things in there like accessibility standards, documentation style guides, commit message formats, or coding standards. For example, "Always include semantic HTML elements in generated markup," or "Never skip heading levels." Remember that `/init` command we just discussed? That's what it helps you generate.

[ALEX]
Now, for teams that use multiple AI tools, or in large monorepos where different subdirectories have different rules, there's `AGENTS.md`. This is an open standard instruction file recognized by multiple AI tools, including Copilot. If you have this file at the root, or nested in subdirectories, it applies. The closest `AGENTS.md` file to your current directory wins.

[JAMIE]
So, `copilot-instructions.md` is for Copilot specifically, but `AGENTS.md` is more universal?

[ALEX]
That's the idea. You generally choose one or the other for your primary project instructions. There's also `CLAUDE.md`, for cross-tool compatibility if your team uses both Copilot and Claude Code. It follows a similar pattern for scope and priority.

[PAUSE]

[ALEX]

[JAMIE]
Like a mini-rulebook for just the Python files, or just the documentation folder?

[ALEX]
Spot on. You create these files, often in a folder like `.github/instructions/`, and you add some metadata at the top called "frontmatter." In this frontmatter, you can specify an `applyTo` glob pattern. For instance, `applyTo: "**/*.py"` would make these instructions only active when you're working with Python files.

[ALEX]
So, you could have a file called `test-file-standards.instructions.md` with an `applyTo` glob of `**/*.test.{js,ts}`. Inside, you'd define rules like, "Use `describe`/`it` structure," or "Test names must describe behavior."

[JAMIE]
That's incredibly powerful! So I can tell Copilot, "When you're helping me with a test file, remember these specific testing conventions."

[ALEX]
Exactly. These instructions can be triggered automatically when a matching file is in context, or Copilot can detect from the file's description that it's relevant to your task, or you can even manually select them from the chat panel.

[ALEX]
And for large organizations, there are "organization-level instructions." These are defined by GitHub Enterprise or Teams administrators and apply to all repositories in the organization. They act as a base layer of instructions, covering things like company-wide security policies or legal disclaimers.

[JAMIE]

[ALEX]
Yes, exactly. It's the lowest priority, though, meaning your workspace-level and user-level instructions can override or add to them if there's a conflict. All non-conflicting instructions are combined.

[ALEX]
Just a quick note: there were "settings-based instructions" where you could put instructions directly into your VS Code `settings.json` for specific tasks like code generation or test generation. However, these are now deprecated, and the file-based approach is much preferred for new work.

[ALEX]
And this leads us nicely into the broader concept of "GitHub Agentic Workflows." This is a technical preview feature that takes AI agents out of your local editor and runs them as part of GitHub Actions pipelines in the cloud.

[JAMIE]
Agents in the cloud? What does that even mean?

[ALEX]
Think of it this way: instead of you typing prompts to Copilot in VS Code, you're now writing automation goals in plain Markdown files in your repository. Then, a command-line interface, `gh aw`, compiles these Markdown files into standard GitHub Actions workflows. The AI agent then interprets your natural language description and executes the task autonomously in the cloud.

[ALEX]
The key differentiator here is that these agents can work autonomously, without you needing to be present in VS Code. They can be triggered by events like a new issue being opened, a push to your main branch, or even on a daily schedule.

[JAMIE]
So, it's like a robot assistant that lives in my GitHub repository and can perform tasks automatically based on instructions I give it in Markdown?

[ALEX]
Exactly! It's an automated robot assistant, but with very strict safety protocols. This enables things like auto-triaging, labeling, and assigning new issues, keeping your README and documentation in sync with code changes, or even generating automated refactoring pull requests.

[ALEX]
A critical part of this is the "security model" with "safe outputs." These workflows run read-only by default. Any write operations, like creating an issue or commenting on a pull request, must be explicitly declared in the frontmatter of your Markdown workflow file under `safe-outputs`. This acts as a security guardrail, ensuring the AI agent can only perform pre-approved, sanitized GitHub operations. It cannot push arbitrary code or delete branches unless you've explicitly configured it to do so through these safe outputs.

[ALEX]
So, you define your goal in a Markdown file, `gh aw` compiles it, and then the AI agent performs the task in GitHub Actions, with strict security measures in place.

Finally, Jamie, a crucial workflow for our listeners, and truly for anyone interacting with Copilot, is the "Accessible View workflow."

[JAMIE]

[ALEX]
You've hit on exactly why the Accessible View, accessed with Alt plus F2, is so vital. Copilot Chat responses stream in token by token, which can indeed fragment screen reader announcements. The Accessible View gives you a complete, static, and properly structured version of the response.

[ALEX]
Here's the recommended workflow for every Copilot interaction:
First, open Copilot Chat with Control plus Shift plus I.
Second, type your prompt and press Control plus Enter to send it.
Third, immediately press Alt plus F2. The Accessible View will open and the response will stream directly into it, in real-time.

[JAMIE]
Immediately? Even before Copilot finishes generating the response?

[ALEX]
Yes, exactly. You don't have to wait. This ensures you get the full response in one readable pane, without the live region updates interrupting or overlapping. You can navigate it line by line with your arrow keys, and headings, code blocks, and lists are all properly structured. When you're done, just press Escape to close it.

[ALEX]
This workflow is also critical for inline suggestions. When a multi-line ghost text suggestion appears, don't accept it immediately. Press Alt plus F2 to open the Accessible View, read the complete suggestion at your own pace, then press Escape to close the view, and finally Tab to accept the suggestion or Escape again to dismiss it.
